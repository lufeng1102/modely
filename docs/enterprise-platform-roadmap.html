1. 平台核心定位

当前 modely-ai 更偏向 CLI/API 工具，已经具备跨平台下载、搜索、文件列表、资源比较、缓存管理、扫描、评分、审计、锁文件，以及 Hugging Face / ModelScope / GitHub / Kaggle 支持。企业级升级的核心，是从“工具”变成“服务平台”，从“按需下载”变成“自动同步”，从“本地缓存”变成“企业资产仓库”，从“资源工具”变成“AI 资产治理中台”。

推荐定位：企业级 AI 开源资产中台。统一同步、缓存、治理、分发和审计来自 Hugging Face、ModelScope、Kaggle、GitHub 等开源生态的模型、数据集、工具仓库，为企业内部研发、算法、平台、安全、合规团队提供可信、可控、可追溯的 AI 资产供应链入口。

从工具升级为服务平台

CLI/API 继续保留，同时增加 Server、Web UI、管理后台、REST/gRPC API。

从按需下载升级为自动同步

支持定时同步、规则化同步、增量同步、版本追踪和失败恢复。

从本地缓存升级为企业资产仓库

统一存储、权限控制、元数据管理、生命周期管理和审计追踪。

从资源工具升级为资产治理中台

安全扫描、License 合规、Secret 检测、模型风险评级和使用审批。

从外网直连升级为内部可信分发

内部镜像、Signed URL、服务账号、下载审计和缓存命中优化。

从单点能力升级为平台生态

对接训练平台、推理平台、MLflow、DVC、CI/CD 和企业 SSO。

一、资源同步中心

这是企业级平台最核心的新增能力。目标是从“用户临时下载”升级为“平台自动发现、同步、镜像、校验并准入”。

3.1 多源同步

Source资源类型说明
Hugging Face模型、数据集、Spaces、文件优先支持模型和数据集，后续可扩展 Space 元数据。
ModelScope模型、数据集支持轻量 HTTP 与官方 SDK 后端。
Kaggle数据集、模型、竞赛数据需要企业凭证管理和 API 配额控制。
GitHub工具仓库、模型代码、数据处理工具在当前产品语义中 GitHub 资源统一作为 tool。
GitLab / Gitee / 内部 Git企业内部工具仓库适合企业私有工具和模型工程仓库。
S3 / OSS / MinIO内部对象存储资产用于导入已有内部模型、数据集或归档文件。
自定义 HTTP Registry第三方模型源通过插件化 Source Provider 扩展。

3.2 同步模式

触发模式

手动同步定时同步Webhook 触发API 触发

数据范围

增量同步全量同步指定版本同步指定 tag/revision/branch

文件策略

指定文件类型include/exclude只同步元数据只同步 manifest

下载策略

延迟下载文件按需 materialize限速并发控制 
modely-ai sync add hf://models/Qwen/Qwen2.5-7B-Instruct \
  --schedule "0 2 * * *" \
  --include "*.safetensors,config.json,tokenizer.*" \
  --policy enterprise-default

3.3 同步规则

  • 按组织同步：同步某个 Hugging Face org 下所有模型。
  • 按标签同步：同步带 text-generation、embedding 标签的模型。
  • 按热度同步：下载量超过阈值。
  • 按 License 同步：只同步 Apache-2.0 / MIT / BSD。
  • 按文件大小同步：限制最大单文件大小。
  • 按资源类型同步：model / dataset / tool。
  • 按更新时间同步：只同步最近 N 天更新资源。
  • 按安全策略同步：不允许 remote code 的资源。

五、安全与合规治理

这是从工具变成企业平台时必须补强的部分。企业内部外部资产准入需要可策略化、可阻断、可审计。

7.1 License 合规

  • SPDX 标准化。
  • License 自动识别。
  • 多 License 文件合并判断。
  • License 风险等级。
  • 商用风险提示。
  • Copyleft 风险提示。
  • 未声明 License 阻断。
  • License 策略模板。
license:
  allow:
    - Apache-2.0
    - MIT
    - BSD-3-Clause
  warn:
    - CC-BY-NC-4.0
  block:
    - GPL-3.0
    - AGPL-3.0
    - unknown

7.2 Secret 检测

针对 GitHub 工具仓库、模型代码、配置文件，需要检测 API Key、Token、私钥、密码、云厂商凭证、数据库连接串、内部地址泄露。

7.3 Remote Code 风险

模型和工具资源中常见风险包括 trust_remote_code=True、自定义 Python module、setup.py、shell script、notebook、pickle、torch .bin、不可信动态 import。

  • remote code warning。
  • 静态代码扫描。
  • 危险函数检测。
  • 可执行文件检测。
  • pickle 风险提示。
  • 高风险资源审批。

7.4 数据集合规

  • PII 检测。
  • 敏感字段识别。
  • 样本抽检。
  • 数据来源记录。
  • 数据用途限制。
  • 数据集 License 检查。
  • 数据集脱敏状态记录。

六、扫描、评分与质量评估

当前已有 scan、score、compare 等能力，企业化后需要增强为标准化治理能力，并统一报告输出。

8.1 模型评分

元数据完整性、README / model card 完整度、License 清晰度、下载热度、更新时间、文件完整性、权重格式、框架兼容性、安全风险、企业内部使用次数、是否通过审批。

8.2 数据集评分

数据说明完整度、License、数据量、文件格式、Schema 信息、敏感信息风险、质量报告、更新频率、可复现性。

8.3 工具仓库评分

README、License、Star / Fork / Issue 活跃度、最近更新时间、依赖风险、Secret 风险、可执行脚本风险、CI 状态、Release 状态。

8.4 报告输出

需要统一支持 JSON、Markdown、CSV、HTML、SARIF，以及可选 CycloneDX / SBOM。

modely-ai scan hf://models/org/model --format sarif
modely-ai audit export --format csv
modely-ai score hf://models/org/model --format json
